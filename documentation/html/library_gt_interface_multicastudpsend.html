Multicast UDP Send

Library

GenericTarget / Interface / Multicast UDP Send

Description

Send a UDP message to a specified destination. The communication is handled via a global internal socket that must be bound to a unique port. All UDP blocks access that internal socket via the port, which is specified under the socket tab. During simulation via Simulink the port should be globally unique among all UDP blocks, because there's no global internal socket but a local one for each block.

Input Ports

destination

A 5-by-1 vector denoting the destination address, where the first four values indicate the destination IPv4 address (LSByte) and the fifth value indicates the destination port. Internally, the first four values are masked by 0x00FF.
Data Type: uint16

bytes

A vector containing the message that should be sent.
Data Type: uint8

length

The number of bytes that should be sent. The length must not be greater than 65507. If the length is zero, then no UDP message is sent.
Data Type: uint32

Output Ports

result

This output port is enabled, when checking the option Show result output port via the block mask. The value represents the number of bytes that have been sent or a negative value in case of errors.
Data Type: int32

lastErrorCode

This output port is enabled, when checking the option Show lastErrorCode output port via the block mask. The value indicates the last socket error code from sending a message. 0 means no error. The actual meaning of non-zero values depend on the operating system.
Data Type: int32

Socket

A socket represents the programming interface from the application to the network hardware. During the initialization phase of the model, all UDP blocks register with the generic target core, which manages all sockets in a hash table. A socket is uniquely identified by its local port. This port serves as the key value for the internal hash table. If several UDP blocks register with the same port, then these blocks are linked to the same socket. All UDP blocks that use the same local port thus always access the same socket. All other settings that can be made in a block mask always refer to the socket that belongs to the specified port. If, for example, two different interface addresses are specified for a socket, then the interface address that does not correspond to the null address is selected. If there are several such interface addresses, the last of these addresses that registers with the generic target core in the generated code is used.

Note

Only one UDP receive block should be used for a socket. At the same time several UDP Send blocks can be used for the same socket. While the UDP receive block outputs all received messages from the buffer in the executed sample step, a message is sent immediately when the UDP send block is executed.

Multicast Interface Address

If no interface is specified, e.g. [0,0,0,0], then the interface leading to the default route is used by the operating system. If different interfaces are specified for the same socket, the interface that appears last in the generated code and is not null is used.

Port Range

The port range [0,1023] is reserved for system ports and should not be used. The range [1024,49151] is defined for user ports. The remaining range [49152,65536] is reserved for dynamic ports. However, not all operating systems follow this standardization. Under certain circumstances, ports in the dynamic ports range may not be able to be used explicitly because the operating system prevents this. If a port cannot be bound, then this port is either in use by another program or the operating system prevents its use.

Application Port

Keep in mind that the target application uses its own port (portAppSocket) to prevent the target from being started multiple times. This port is set via the GT.GenericTarget class and cannot be used again in the model.

Socket Priority

The socket priority is only available if the target application runs on a linux operating system. On windows machines, this value has no effect. This value must be in range [0,6] and sets the protocol-defined priority for all packets to be sent on this socket. Linux uses this value to order the networking queues: packets with a higher priority may be processed first depending on the selected device queueing discipline. If different socket priorities are specified for the same socket (same port), then the largest value is selected.

Group Address

During the initialization phase of the model, each UDP block registers with the generic target core. A group address is registered for each multicast socket. If multiple group addresses are registered for the same socket, then the address that is executed last in the generated code is taken. A multicast socket then automatically joins the registered group. The receiving block can thus receive multicast messages. If messages are to be sent to a specific group address, then this must be specified as the destination address via the input port. In principle, a message can also be sent to a multicast group with a unicast socket if the group address is set as the destination address. However, the TTL value is only set when using multicast sockets, so the Multicast UDP Send block should be used to send multicast messages.

Time-to-live (TTL)

When sending multicast messages a TTL value is set which corresponds to a hop counter. Each time one of these packets passes through a multicast routing device, the TTL setting decrements by 1. If it falls below a threshold, the multicast routing device drops the message. If several TTL values are registered for a socket in the target application, then the highest TTL value is used.
