Multicast UDP Receive

Library

GenericTarget / Interface / Multicast UDP Receive

Description

Receive UDP messages from a multicast group. The communication is handled via a global internal socket that must be bound to a unique port. All UDP blocks access that internal socket via the port, which is specified under the socket tab. During simulation via Simulink the port should be globally unique among all UDP blocks, because there's no global internal socket but a local one for each block.

Output Ports

source

A 5-by-M vector denoting the source address, where M denotes the number of messages in the buffer. The first four values in a column indicate the source IPv4 address (LSB) and the fifth value in a column indicates the source port.
Data Type: uint16

bytes

An N-by-M matrix containing the messages that have been received, where N indicates the buffer size for a single message and M denotes the number of messages in the buffer.
Data Type: uint8

length

An M-by-1 vector indicating the actual length of the corresponding message. M denotes the number of messages in the buffer.
Data Type: uint32

timestamp

An M-by-1 vector indicating the timestamp in seconds when the corresponding message has been received by the receiver thread. This time corresponds to the Model Execution Time.
Data Type: double

numReceivedMessages

The number of messages that have been received.
Data Type: uint32

numDiscardedMessages

This output port is enabled, when checking the option Show numDiscardedMessages output port via the block mask. The value represents the number of messages that have been discarded.
Data Type: uint32

lastErrorCode

This output port is enabled, when checking the option Show lastErrorCode output port via the block mask. The value indicates the last socket error code from sending a message. 0 means no error. The actual meaning of non-zero values depend on the operating system.
Data Type: int32

Socket

A socket represents the programming interface from the application to the network hardware. During the initialization phase of the model, all UDP blocks register with the generic target core, which manages all sockets in a hash table. A socket is uniquely identified by its local port. This port serves as the key value for the internal hash table. If several UDP blocks register with the same port, then these blocks are linked to the same socket. All UDP blocks that use the same local port thus always access the same socket. All other settings that can be made in a block mask always refer to the socket that belongs to the specified port. If, for example, two different interface addresses are specified for the same socket, then the generic target core reports an error and the software terminates.

Note

Only one UDP receive block should be used for a socket. At the same time several UDP Send blocks can be used for the same socket. While the UDP receive block outputs all received messages from the buffer in the executed sample step, a message is sent immediately when the UDP send block is executed. If multiple receivers are registered for the same port, then the generic target core reports an error and the software terminates.

Interface

The network interface used to join the multicast group is set in the receive block mask. The interface can be specified either as IPv4 addresse or as name. If no interface is specified, e.g. [0,0,0,0], then all interfaces are used to join the multicast group.

Port Range

The port range [0,1023] is reserved for system ports and should not be used. The range [1024,49151] is defined for user ports. The remaining range [49152,65536] is reserved for dynamic ports. However, not all operating systems follow this standardization. Under certain circumstances, ports in the dynamic ports range may not be able to be used explicitly because the operating system prevents this. If a port cannot be bound, then this port is either in use by another program or the operating system prevents its use.

Application Port

Keep in mind that the target application uses its own port (portAppSocket) to prevent the target from being started multiple times. This port is set via the GT.GenericTarget class and cannot be used again in the model.

Buffering

Each socket has its own thread in which the data is received. All received messages are collected in a buffer. The maximum number of messages is set with the maxNumMessages parameter. Each message in the buffer has a maximum number of bytes, which is set with the rxBufferSize parameter. If the buffer is full and more messages are received, then either the received message or the oldest message is deleted from the buffer, depending on which rule is set with the bufferStrategy parameter. In each sampling step in which the receive block is executed, the entire buffer with all messages is read out and made available at the outputs of the block. The buffer is then cleared. The actual number of received messages is defined by the output numReceivedMessages. The numDiscardedMessages output indicates how many messages were discarded, for example because more messages were received than could be stored in the buffer.

Group Address

During the initialization phase of the model, each UDP block registers with the generic target core. A group address is registered for each multicast socket. If multiple group addresses are registered for the same socket, then the generic target core reports an error and the software terminates. A multicast socket then automatically joins the registered group. The receiving block can thus receive multicast messages. If messages are to be sent to a specific group address, then this must be specified as the destination address via the input port. In principle, a message can also be sent to a multicast group with a unicast socket if the group address is set as the destination address. However, the TTL value is only set when using multicast sockets, so the Multicast UDP Send block should be used to send multicast messages.
